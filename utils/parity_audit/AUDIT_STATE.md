# Parser parity audit — state of knowledge

Single source of truth for the Python-vs-Rust parser parity campaign on this
branch. Read this before resuming parity work: it records what has been
verified equal (and at what strictness), every bug found, what remains open,
and how to re-run each instrument.

The three parsers under comparison:

1. **Python parser** (`sqlfluff.core.parser.Parser`) — the reference.
2. **RustParser, default path** — Rust core match + Python
   `_convert_rs_match_result` → `MatchResult.apply()`.
3. **RustParser, native-AST path** — Rust core match + fused
   `_apply_rs_match_result` (env `SQLFLUFF_RS_NATIVE_AST` / `set_native_ast`).

The parity bar throughout: **byte-level** — trees with position markers,
`stringify()` bytes (including UnparsableSegment "Expected:" messages), raw
round-trip, exception type/message/position attributes, parse-node
accounting, logging, CLI output.

    (`SELECT straße(1)` postgres/duckdb) (`60c0b5f`)

20. ~600 dangling keyword refs → 371 keyword registrations across 16

    grammar fixes (`60c0b5f`)
21. (housekeeping) three-way sweep strictness upgrades along the way

Open, each pinned by a **strict xfail** (fails CI the moment someone fixes it):

- **escape_replacements truncated to first pair** (`build_lexers.py` takes
  `[0]`; whole Rust pipeline is singular). Flink loses its `\'` pair.
  → `lexer_cases.yml:flink_multiple_escape_replacements`
- **Depth-guard SQLParseError position**: Rust anchors line/pos, Python
  raises unanchored from `ParseContext.deeper_match`.
  → `known_divergences.yml:depth_guard_error_position`
- **max_parse_nodes double enforcement**: Rust core counts internal match
  nodes (limit 50 for a simple SELECT), Python counts materialized nodes
  (46) — limits in the window behave differently per engine.
  → `special_cases_test.py::test__parity__vs_python_max_parse_nodes_threshold`
- **Four rust-core matching/recovery gaps** (minimized repros embedded):
  sqlite `'Nothing here.'` expected-message; materialize `{`-cross silent
  recovery (Python raises); tsql `CAST(0X0 uniqueidentifier)` structure;
  materialize webhook overlapping-match ValueError.
  → `known_divergences.yml` (four cases)
- **Overlapping child-match emissions** (bracketed/script recovery, exasol +
  materialize) → `invariants.yml:{materialize_webhook_overlap,exasol_script_overlap}`
- **Error-recovery iteration blowup** (found 2026-07 when the edge battery
  gained the python-vs-rust leg): an unclosed 30-deep nested `CASE` burns
  the Rust core's 3,000,000 iteration cap and panics, where Python recovers
  an unparsable tree in ~0.1s.
  → `native_edge_cases.yml:nested_case_unclosed`
- Pre-existing: snowflake `-}` hardcoded bracket set
  (`bracket_recovery.yml:stray_closing_bracket_dialect_bracket_set`); lexer
  unlexable error message format
  (`lexer_cases.yml:unlexable_error_message_format`).

Known-but-accepted (documented, no fix planned):

- 12 RegexParser pattern-level differences hinging solely on U+0130 (İ):
  unreachable through any dialect lexer.
- Transient node-counter value inside an already-aborted parse.
- RecursionError boundary at exotic `sys.setrecursionlimit` values.
- `lint --format json` `"timings"` object is wall-clock nondeterministic
  (normalize it in any byte-diff).

## Caveats recorded for maintainers

- The 371 keyword additions light up inherited grammar branches in dialects
  that may not really support those statements (e.g. GRANT variants in
  sqlite/trino). Previously those branches CRASHED Python when routed;
  parsing beats crashing, but dialect-fidelity review is welcome. No existing
  fixture changed its parse (full dialect suite stayed green).
- vertica `MERGEOUT OneOf("1","2")` → `Ref("IntegerSegment")` loosens the
  accepted values slightly (any integer).
- The generated Rust dialect tables are **gitignored** and rebuilt by
  `sqlfluffrs_dialects/build.rs` (mtime-triggered) or `python
  utils/rustify.py build`. Any codegen change silently reshapes them on the
  next build — which is how bug #4 stayed invisible.

## How to resume

All instruments live in this directory (run from here, not the repo root —
the `sqlfluffrs/` source dir shadows the installed extension):

```
python pyrs_harness.py                 # strict py-vs-rs corpus sweep
python lexer_diff.py                   # lexer differential
python table_check.py [dialects...]    # table↔grammar semantic checks
python invariant_fuzz.py [rounds]      # match-result invariant fuzzing
python regex_diff.py                   # regex accept-decision differential
                                       # (build the rxdiff helper crate it
                                       # documents: regex=1.12.3,
                                       # fancy-regex=0.17.0, ^(?:pat)$ wrap,
                                       # uppercase-when-ci)
```

Fuzz seeds are fixed (mutation 1234, char 777, soup 31337, invariant
90000+round) so every historical finding regenerates deterministically.
The committed guards are the data-driven suite under
`test/core/parser/parity/` (drivers) + `test/fixtures/parity/*.yml` (case
corpus), documented in `test/fixtures/parity/README.md`. Add new parity
regressions as YAML cases there, not as bespoke Python tests. The bare
`*.yml` names throughout this document refer to that corpus. Non-parity
RustParser unit tests (config, iteration limits, profiling, arena
navigation) remain in `test/core/parser/rust_parser_test.py`.
